# Overwatch 英雄匹配问卷项目计划（Plan + Research）

> 目标：先对齐真实需求，再实施；保证切换模型后也能稳定交付。

## 0. 背景与问题定义

我们要做一个「可发布」的网页问卷，帮助玩家判断在《守望先锋》中更适合的英雄/定位，从而提升体验与留存。

当前风险：
- 需求还不够明确（娱乐向 vs 竞技向、新手 vs 高玩）
- 结果解释可能过于主观
- 模型切换后输出风格漂移，导致实现不一致

## 1. 真实需求确认（先问清）

### 1.1 产品目标（必须确认）
- 核心目标：
  - A. 娱乐分享（轻量有趣）
  - B. 训练建议（有一定科学性）
  - C. 分流推荐（引导玩家选位）
- 成功指标（至少选一个）：
  - 完成率 >= X%
  - 分享率 >= X%
  - 玩家自评“推荐合理” >= X%

### 1.2 目标用户
- 新手 / 回归玩家 / 中高分段玩家
- 单排为主还是开黑为主
- PC/主机平台差异是否要考虑

### 1.3 结果粒度
- 只推荐「定位」(Tank/DPS/Support)
- 推荐到「英雄池」（每类3-5个）
- 是否输出“主练+副练”与训练建议

### 1.4 发布与运维
- 发布渠道：GitHub Pages / Vercel / Netlify / 私有站点
- 是否需要埋点分析（GA/Umami）
- 是否需要多语言（中文优先，后续英文）

## 2. Research 计划

### 2.1 内容研究
- 收集官方英雄设计定位与技能职责
- 梳理玩家常见风格维度：
  - 机械操作（准度/追踪/反应）
  - 战术意识（站位/CD管理/开团时机）
  - 团队偏好（沟通/保人/资源交换）
  - 风险偏好（保守/激进）

### 2.2 问卷方法研究（轻量）
- 题型：Likert 5分量表
- 题量：10题（MVP）→ 20题（Pro）
- 质量控制：
  - 避免双重问题
  - 正反向题平衡（后续版本）
  - 结果解释可读且可执行

### 2.3 校准研究
- 用 20-50 份样本做初步校准
- 观察“推荐合理性”反馈
- 调整权重和阈值（而非只改文案）

## 3. 功能范围（MVP）

### 3.1 必做
- 单页问卷（移动端优先）
- 自动评分与类型判定
- 结果页：定位 + 英雄推荐 + 训练建议
- 重置与复测

### 3.2 可选
- 结果分享卡片
- URL 参数预填
- 基础埋点（开始率/完成率/分享点击）

### 3.3 暂不做
- 账号系统
- 后端数据库
- 复杂个性化推荐模型

## 4. 评分模型（可解释）

- 维度分：DPS / Tank / Support / Flanker
- 每维 3 题，满分 15
- 输出规则：
  1. 主维度最高者为主推荐
  2. 若 Flanker >= 阈值（如12）且非最高，给副修建议
  3. 分差过小时提示“双定位适配”

> 后续可升级：按玩家反馈做权重校准（v1.1）

## 5. 验收标准（DoD）

- 功能：问卷可完整填写、可出结果、逻辑可复现
- 体验：手机端可用，首屏加载快，文案清晰
- 稳定：无阻断错误，输入校验完整
- 可移交：任何模型接手都能按文档复现

## 6. 模型切换稳定协议（关键）

为避免不同模型产出漂移，后续执行统一遵循：

1. 先读本文件 `PLAN.md`
2. 先输出“需求确认清单”，用户确认后再编码
3. 代码变更需匹配本计划的范围与DoD
4. 每次迭代更新：
   - `CHANGELOG.md`（改了什么）
   - `DECISIONS.md`（为什么这样改）
5. 禁止跳过 research 直接堆功能
